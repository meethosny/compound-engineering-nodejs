Getting Started with Compounding Engineering

Five minutes from now, you'll run a single command that spins up a tiered set of reviewer personas—each with a different specialty—to review your pull request in parallel. Security, performance, architecture, correctness, all happening at once. That's the plugin: 29 self-contained skills, no standalone agents to manage, and no MCP servers to configure. Let's get you set up.

Installation

Prerequisites

  • Claude Code installed and configured
  • A GitHub account (for marketplace access)
  • Node.js 18+ (for MCP servers)

Step 1: Add the Marketplace

Think of the marketplace as an app store. You're adding it to Claude Code's list of places to look for plugins:

claude /plugin marketplace add https://github.com/meethosny/compound-engineering-nodejs

Step 2: Install the Plugin

Now grab the plugin itself:

claude /plugin install js-compound-engineering

Step 3: Verify Installation

Check that it worked:

claude /plugin list

You'll see js-compound-engineering in the list. If you do, you're ready. Then run /js-ce-setup to check your environment and install recommended tools.

One-step setup

The plugin bundles no MCP servers and no standalone agents, so there's nothing extra to wire up. Run /js-ce-setup once in any project and it checks your environment, installs recommended tools, and bootstraps config.

Quick Start

Let's see what this thing can actually do. I'll show you three workflows you'll use constantly:

Run a Code Review

This is the big one. Type /js-ce:code-review and watch the skill spawn its always-on and conditional reviewer personas:

# Review a PR by number
/js-ce:code-review 123

# Review the current branch
/js-ce:code-review

# Review a specific branch
/js-ce:code-review feature/my-feature

Point a Skill at a Focused Task

The plugin is agentless—specialists run inside skills, not as separate agents you call. Scope a skill to one problem and it dispatches the right persona for you:

# Diagnose a failing test with the debug loop
skill: js-ce-debug "The checkout total test is flaky"

# Simplify code you just wrote
skill: js-ce-simplify-code

# Get a decisive verdict on a library or pattern
skill: js-ce-pov "Should we adopt Drizzle over Prisma here?"

Invoke a Skill

Skills package the plugin's workflows and domain knowledge. When you need a specific workflow:

# Explain a concept or a window of recent work
skill: js-ce-explain

# Turn a vague idea into a requirements doc
/js-ce:brainstorm

# Plan a multi-step task
/js-ce:plan "Add OAuth integration"

Configuration

MCP Servers

Nothing to configure. The plugin ships zero MCP servers, so there's no endpoint to wire up and nothing that can fail to auto-load. If you want to add your own server—say, a documentation service—see the MCP Servers page for a copy-paste config.

Environment Variables & Secrets

Most skills need no API keys. Skills that integrate an external service (Slack, GitHub, Proof, and similar) read their settings from a repo-local .js-compound-engineering/config.local.yaml and prompt you when something is missing. Run /js-ce-setup to check what a given project needs.

The Compounding Engineering Philosophy

Every unit of engineering work should make subsequent units of work easier—not harder.

Here's how it works in practice—the four-step loop you'll run over and over:

1. Plan

Before you write a single line, figure out what you're building and why. Use research agents to gather examples, patterns, and context. Think of it as Google Search meets expert consultation.

2. Delegate

Now build it—with help. Each agent specializes in something (Node.js, security, design). You stay in the driver's seat, but you've got a team of specialists riding shotgun.

3. Assess

Before you ship, run the gauntlet. Security agent checks for vulnerabilities. Performance agent flags N+1 queries. Architecture agent questions your design choices. All at once, all in parallel.

4. Codify

You just solved a problem. Write it down. Next time you (or your teammate) face this, you'll have a runbook. That's the "compounding" part—each solution makes the next one faster.

The Agentless Model

There are no standalone agents to install or call. The specialists you'd expect—security reviewer, performance reviewer, framework-docs researcher—live inside the skills that use them, as prompt-only personas. When a skill needs an expert opinion, it dispatches a generic subagent with that persona's prompt.

Why agentless?

  • Self-contained skills. Each skill carries its own personas under references/personas/, so there's no separate agent registry to keep in sync.
  • Portable. A skill converts to other agent tools as a single unit—no agent-registration step.
  • Nothing to memorize. You run a skill; it selects and dispatches the right specialists for you.

Where specialists get dispatched

Skill Specialists it dispatches
/js-ce:code-review correctness, maintainability, testing, and project-standards reviewers, plus conditional personas (security, performance, migrations, CLI, async-UI, Swift) chosen by what the diff touches
js-ce-doc-review coherence, feasibility, design, product, scope, security, and adversarial document review lenses
/js-ce:plan research personas for framework docs, git history, prior learnings, and the open web

How the agentless model works

Using Commands

Commands are macros that run entire workflows for you. One command can spin up a dozen agents, coordinate their work, collect results, and hand you a summary. It's automation all the way down.

Commands are skills now. Every former /command was migrated into a skill and works identically as a slash command. See the commands note for the mapping.

Running Commands

# Core workflow (colon-namespaced slash commands)
/js-ce:plan
/js-ce:code-review 123
/js-ce:work
/js-ce:compound

# Other skills (hyphenated names)
/js-ce-debug
/js-ce-commit
/js-ce-strategy

The Review Workflow

Let me show you what happens when you run /js-ce:code-review. Here's the sequence:

  1. Detection - Figures out what you want reviewed (PR number, branch name, or current changes)
  2. Persona selection - Runs always-on reviewers (correctness, maintainability, testing, project standards) plus conditional personas chosen by what the diff touches
  3. Parallel execution - Dispatches the selected reviewer personas as subagents simultaneously
  4. Merge & dedup - Combines findings, removes duplicates, and confidence-gates them
  5. Synthesis - Sorts findings by severity (P1 = blocks merge, P2 = should fix, P3 = nice-to-have)
  6. Summary - Hands you a readable report with action items

Using Skills

Here's the difference: agents are who does the work, skills are what they know. When you invoke a skill, you're loading a reference library into Claude's context—patterns, templates, examples, workflows. It's like handing Claude a technical manual.

Invoking Skills

# In your prompt, reference the skill by its hyphenated name
skill: js-ce-debug

# Or ask Claude to use it
"Use the js-ce-simplify-code skill to clean up this module"

Skill Structure

Peek inside a skill directory and you'll usually find:

  • SKILL.md - The main instructions (what Claude reads first)
  • references/ - Deep dives on concepts and patterns
  • templates/ - Copy-paste code snippets
  • workflows/ - Step-by-step "how to" guides
  • scripts/ - Actual executable code (when words aren't enough)

Code Review Workflow Guide

You'll spend most of your time here. This workflow is why the plugin exists—to turn code review from a bottleneck into a superpower.

Basic Review

# Review a PR
/js-ce:code-review 123

# Review current branch
/js-ce:code-review

Understanding Findings

Every finding gets a priority label. Here's what they mean:

  • P1 Critical - Don't merge until this is fixed. Think: SQL injection, data loss, crashes in production.
  • P2 Important - Fix before shipping. Performance regressions, N+1 queries, shaky architecture.
  • P3 Nice-to-Have - Would be better, but ship without it if you need to. Documentation, minor cleanup, style issues.

Acting on Findings

In interactive mode, /js-ce:code-review applies safe, verified fixes for you and commits them when the working tree is clean (it never pushes). In mode:agent it reports findings only and leaves the changes to you. For refinement passes after the bugs are gone, reach for the quality-focused skills: 

# Simplify recently changed code without altering behavior
skill: js-ce-simplify-code

# Resolve open PR review threads in parallel
skill: js-ce-resolve-pr-feedback

Adding Specialists

Because the plugin is agentless, you don't create a standalone agent file—you extend the skill that needs the specialist. Say you want /js-ce:code-review to run a house "api-conventions" reviewer: add its prompt as a persona inside the code-review skill.

Persona File

Drop a plain prompt—no YAML frontmatter, unprefixed filename—under the calling skill's references/ directory:

skills/js-ce-code-review/references/personas/api-conventions-reviewer.md

# You are an API-conventions reviewer.
# Review the diff for breaking changes to public
# routes, response shapes, and versioning...

Then have the skill dispatch a generic subagent with that prompt. Personas are internal to the skill, so there's no agent count to update and no registry to touch. Keep each skill self-contained: if two skills need the same specialist, give each its own copy.

Prefer a personal, reusable agent?

That's a Claude Code feature, separate from this plugin: drop a definition in .claude/agents/ (this project) or ~/.claude/agents/ (all your projects). The plugin itself stays agentless.

Creating Custom Skills

Skills are heavier than agents—they're knowledge bases, not just prompts. You're building a mini library that Claude can reference. Worth the effort for things you do repeatedly.

Skill Directory Structure

my-skill/
  SKILL.md           # Main skill file (required)
  references/        # Supporting documentation
    concept-one.md
    concept-two.md
  templates/         # Code templates
    basic-template.md
  workflows/         # Step-by-step procedures
    workflow-one.md
  scripts/           # Executable scripts
    helper.py

SKILL.md Format

---
name: my-skill
description: Brief description shown when skill is invoked
---

# Skill Title

Detailed instructions for using this skill.

## Quick Start
...

## Reference Materials
The skill includes references in the `references/` directory.

## Templates
Use templates from the `templates/` directory.

Get Help Building Skills

Use the skill-creator skill (or your platform's skill-authoring helper) to scaffold a new SKILL.md, then follow the plugin's skill compliance checklist for frontmatter and reference conventions.
